# Internationalization (i18n) & Localization (l10n) — Open on4net (O₂N)

> **فایل:** 00_VISION/08-i18n-l10n.md  
> **نسخه:** 1.0 | **تاریخ:** July 2026

---

## ۱) هدف

O₂N باید از روز اول چندزبانه باشد، چون:
- کاربران شرکت‌ها ممکن است فارسی/انگلیسی/ترکی/عربی استفاده کنند
- خروجی Agentها باید به زبان سازمان باشد
- UI باید RTL/LTR را پشتیبانی کند

---

## ۲) زبان‌ها (MVP)

پیشنهاد MVP:
- `fa-IR` (RTL)
- `en-US` (LTR)
- `tr-TR` (LTR)

فاز بعد:
- `ar` (RTL)

---

## ۳) i18n در UI (Next.js)

الزام‌ها:
- route-based locale (مثلاً `/fa/...`, `/en/...`)
- یک منبع ترجمه‌ی versioned (JSON)
- direction switching (RTL/LTR) + فونت‌های مناسب
- number/date formatting بر اساس locale
- همه labelها و error messages از i18n system بیایند (نه hardcode)

---

## ۴) زبان در Agent/Prompt

### ۴.۱) انتخاب زبان خروجی
قانون:
- language preference در سطح org/workspace/user ذخیره می‌شود
- prompt injection نباید اجازه دهد زبان و policyها override شوند

### ۴.۲) Memory و زبان
- در ingestion، هر سند `source_language` و `content_language` می‌گیرد
- search باید multilingual باشد (یا embedding multilingual یا translate-on-index)

---

## ۵) API Contracts (Locale)

Headerهای پیشنهادی:
- `Accept-Language` (مثلاً `fa-IR`)
- `X-Timezone` (مثلاً `Europe/Istanbul`)

کاربرد:
- پیام‌های خطا/سیستم
- گزارش‌ها و فرمت تاریخ/عدد

---

## ۶) Marketplace i18n

برای plugin listings:
- `name` و `description` باید multi-locale شوند (در metadata)
- UI marketplace بر اساس زبان کاربر نمایش دهد

---

> **خلاصه:** چندزبانه بودن در O₂N فقط UI نیست؛ زبان باید در prompt، memory ingestion و marketplace metadata هم طراحی شود.
