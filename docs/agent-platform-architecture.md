# معماری پلتفرم‌های «Agent Builder» (مثل AutoGPT، CrewAI، Dify، Flowise)

این سند برای کسی نوشته شده که برنامه‌نویسی بلده اما نمی‌دونه سیستم‌هایی که به کاربر اجازه می‌دن یک ایجنت هوش مصنوعی بسازه، بهش «مهارت» (tool/skill) بده، و بذاره کارها رو خودکار انجام بده (مثل ایده‌ی Hermes Agent / OpenClaw که مطرح کردی) از داخل چطور کار می‌کنن.

نکته: من اطلاعات دقیق و تأییدشده‌ای از معماری داخلی محصولات خاصی به اسم «Hermes Agent» یا «OpenClaw» ندارم، پس روی الگوی کلی‌ای تمرکز می‌کنم که در ابزارهای شناخته‌شده‌ی این حوزه (AutoGPT، CrewAI، LangGraph، Dify، Flowise، n8n) تکرار می‌شه — همون الگو رو هر پلتفرمی از این جنس، از جمله محصولاتی که اسم بردی، دنبال می‌کنه.

---

## ۱. این سیستم‌ها دقیقاً چه‌کاری انجام می‌دن؟

سه مفهوم پایه:

- **Agent** — یک تنظیمات ذخیره‌شده شامل: یک system prompt/شخصیت، یک مدل زبانی، و یک لیست از مهارت‌هایی که بهش اجازه دسترسی داده شده.
- **Skill / Tool** — یک تابع مشخص با ورودی/خروجی مشخص (مثلاً «جستجوی وب»، «ارسال ایمیل»، «اجرای کد پایتون»، «خواندن یک فایل») که مدل می‌تونه صداش بزنه.
- **Execution Loop (Orchestrator)** — حلقه‌ای که: پیام کاربر رو می‌گیره → به مدل زبانی می‌ده همراه با لیست مهارت‌های مجاز → اگر مدل خواست از یک مهارت استفاده کنه، اون مهارت رو واقعاً اجرا می‌کنه → نتیجه رو به مدل برمی‌گردونه → این چرخه تا رسیدن به جواب نهایی تکرار می‌شه.

این دقیقاً همون مکانیزم «tool use» است که در Claude API و OpenAI API به‌صورت native پشتیبانی می‌شه؛ پلتفرم‌هایی مثل AutoGPT/CrewAI/Dify فقط یک لایه‌ی مدیریتی (UI، ذخیره‌سازی، صف اجرا، امنیت) دور همین مکانیزم می‌سازن.

---

## ۲. معماری کلی — لایه‌های سیستم

```
[ کاربر ]
   │
   ▼
[ Frontend: صفحه‌ی ساخت Agent، انتخاب مهارت، چت با Agent ]
   │  (HTTP/WebSocket)
   ▼
[ Backend API + Auth (چندمستأجری/multi-tenant) ]
   │
   ▼
[ پایگاه‌داده: تعریف Agentها، Skillها، تاریخچه‌ی اجرا ]
   │
   ▼
[ Orchestration Engine — حلقه‌ی اجرای Agent ]
   │            │                    │
   ▼            ▼                    ▼
[ LLM Provider ] [ Tool Executor / Sandbox ] [ Memory / Vector DB ]
                       │
                       ▼
              [ صف اجرا (Queue) برای کارهای طولانی ]
                       │
                       ▼
      [ Observability / Logging / Billing ]
```

هر جعبه در این نمودار یک تصمیم تکنولوژی جداست که در ادامه بازش می‌کنیم.

---

## ۳. لایه به لایه: تکنولوژی، مزیت، عیب

### ۳.۱ Frontend (رابط ساخت Agent و چت)

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Next.js (React)** | یک پروژه برای UI و بخشی از backend (API routes)؛ اکوسیستم بزرگ؛ SSR برای صفحات عمومی | برای اپ‌های خیلی بزرگ ممکنه backend جدا بهتر باشه |
| **React + Vite (بدون Next)** | ساده‌تر، build سریع‌تر، جدایی کامل از backend | باید API/روتینگ سمت سرور رو خودت جدا بسازی |
| **Streaming UI (SSE/WebSocket)** برای نمایش زنده‌ی پاسخ Agent | تجربه‌ی کاربری «زنده» مثل ChatGPT | پیچیدگی مدیریت اتصال، reconnect، state حین قطعی شبکه |

**توصیه:** Next.js + یک اتصال SSE یا WebSocket برای استریم پاسخ Agent؛ برای شروع ساده‌تر از WebSocket است چون یک‌طرفه و روی HTTP معمولی کار می‌کنه.

### ۳.۲ Auth و چندمستأجری (Multi-tenancy)

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Auth.js (NextAuth)** | رایگان، self-hosted، کنترل کامل روی داده‌ی کاربر | باید خودت session/reset password/2FA رو مدیریت کنی |
| **Clerk / Supabase Auth** | آماده، سریع برای MVP، UI آماده برای لاگین | وابستگی به سرویس بیرونی؛ هزینه در مقیاس بالا |
| **Auth0** | قوی برای enterprise، SSO آماده | پیچیده و گرون‌تر برای پروژه‌ی کوچک |

**توصیه:** برای MVP از Supabase (هم Auth هم Postgres یک‌جا) یا Clerk استفاده کن تا وقتت صرف مسئله‌ی اصلی (orchestration) بشه.

### ۳.۳ Backend API / زبان سرور

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Node.js (Express/Fastify/NestJS)** | هم‌زبان با frontend، اکوسیستم npm بزرگ، SDKهای Claude/OpenAI رسمی | مدیریت CPU-heavy tasks (مثل اجرای sandbox) دست‌وپاگیرتره |
| **Python (FastAPI)** | نزدیک‌ترین زبان به اکوسیستم AI/ML (LangChain، LlamaIndex، عمده نمونه‌کدها)؛ async خوب | باید dependency دو زبانه (اگر frontend JS باشه) مدیریت کنی |
| **Go** | کارایی و همزمانی (concurrency) عالی برای orchestrator با هزاران اجرای موازی | اکوسیستم AI/agent در Go هنوز کوچک‌تره، توسعه کندتر |

**توصیه:** اگر با AI/agent تازه‌کاری، **Python + FastAPI** برای بک‌اند orchestration بهترین شروعه چون بیشترین نمونه‌کد، کتابخانه و مستندات این حوزه پایتونی‌ان.

### ۳.۴ موتور Orchestration (حلقه‌ی اجرای Agent)

اینجا قلب سیستمه. سه راه داری:

| رویکرد | مزیت | عیب |
|---|---|---|
| **دست‌ساز (Custom loop با Claude tool-use API)** | کنترل کامل، ساده برای فهمیدن، بدون وابستگی سنگین | باید خودت retry، حافظه، محدودیت گام‌ها رو پیاده کنی |
| **LangGraph** | مدل «گراف حالت» برای جریان‌های پیچیده‌ی چندمرحله‌ای؛ کنترل دقیق روی مسیر اجرا | یادگیری مفاهیم خودش (state, node, edge) لازمه |
| **CrewAI** | مفهوم آماده‌ی «تیمی از Agentها با نقش‌های مختلف»؛ سریع برای دمو | انعطاف کمتر برای رفتارهای خیلی سفارشی؛ لایه‌ی abstraction اضافه |
| **Dify / Flowise (no-code)** | UI بصری drag-and-drop برای ساخت جریان Agent؛ مناسب اگر خودت هم بخوای محصول را no-code بسازی برای کاربر نهایی | اگر بخوای عمیقاً سفارشی‌سازی کنی، به کد زیرینش برمی‌گردی؛ کمتر مناسب به‌عنوان engine داخلیِ محصول خودت |

**توصیه:** برای اینکه واقعاً بفهمی این سیستم‌ها چطور کار می‌کنن، مرحله‌ی اول رو با یک **حلقه‌ی دست‌ساز** روی Claude tool-use بساز (چند صد خط کد بیشتر نیست)؛ بعد اگر جریان‌ها پیچیده‌تر شدن (چند Agent که با هم حرف می‌زنن)، برو سراغ LangGraph.

### ۳.۵ اجرای امنِ مهارت‌ها (Tool Execution & Sandboxing) — **مهم‌ترین بخش امنیتی**

وقتی کاربر بتونه به Agent «مهارت اجرای کد» یا «دسترسی به اینترنت» بده، اون کد باید در یک محیط ایزوله اجرا بشه، وگرنه یک کاربر می‌تونه از طریق Agent خودش به سرور شما یا کاربران دیگه آسیب بزنه.

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Docker Container موقت** | پیاده‌سازی ساده، ابزار فراوان | ایزوله‌سازی سطح kernel ضعیف‌تر از VM؛ نیاز به سخت‌گیری شدید روی network/resource limits |
| **gVisor / Firecracker (microVM)** | ایزوله‌سازی نزدیک به VM واقعی با overhead کم؛ استاندارد صنعتی (AWS Lambda از Firecracker استفاده می‌کنه) | راه‌اندازی و نگه‌داری پیچیده‌تر |
| **WASM sandbox (مثل quickjs/wasmtime)** | سبک، شروع سریع، مناسب اجرای قطعه‌کد کوچک | محدودیت در دسترسی به فایل‌سیستم/شبکه (که همون‌جا هم یک مزیت امنیتیه) |
| **صرفاً محدود به لیست ثابتی از توابع از پیش نوشته‌شده (بدون اجرای کد دلخواه کاربر)** | امن‌ترین حالت، پیاده‌سازی ساده | انعطاف کمتر برای کاربر پیشرفته |

**توصیه‌ی جدی برای فاز اول:** اصلاً اجرای کد دلخواه رو باز نذار. فقط چند مهارت از پیش‌تعریف‌شده (جستجوی وب، درخواست HTTP به فهرست مجاز دامنه‌ها، خواندن/نوشتن در فضای داده‌ی خود کاربر) بده. «اجرای کد دلخواه» رو به فاز بعدی و پشت Firecracker/gVisor موکول کن.

### ۳.۶ اتصال به مدل زبانی (LLM Provider)

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Claude API (Anthropic)** | tool-use قوی و پایدار، پشتیبانی از MCP برای مهارت‌های استاندارد، context window بزرگ | هزینه per-token (قابل مدیریت با انتخاب مدل مناسب هر Agent) |
| **OpenAI API** | اکوسیستم بزرگ، Assistants/Responses API مشابه مفهوم | مشابه از نظر هزینه و محدودیت‌ها |
| **مدل‌های Open-source خودمیزبان (vLLM, Ollama)** | کنترل کامل، بدون هزینه‌ی per-token در مقیاس بالا | نیاز به GPU/زیرساخت، کیفیت معمولاً پایین‌تر از مدل‌های تجاری برتر |

**توصیه:** برای MVP یک provider ثابت (Claude) با قابلیت انتخاب مدل در سطح هر Agent (مثلاً مدل سریع‌تر/ارزان‌تر برای کارهای ساده). اضافه‌کردن provider دوم رو به بعد از اعتبارسنجی محصول موکول کن.

### ۳.۷ حافظه‌ی Agent / RAG (اختیاری، فاز دوم)

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **pgvector (روی همون Postgres)** | یک پایگاه‌داده کمتر برای مدیریت؛ ساده برای شروع | در مقیاس خیلی بالا کندتر از vector DBهای اختصاصی |
| **Pinecone / Qdrant / Weaviate** | بهینه‌شده برای جستجوی برداری در مقیاس بزرگ | زیرساخت/سرویس اضافه برای مدیریت |

**توصیه:** حافظه/RAG رو در MVP اصلاً اضافه نکن؛ فقط وقتی لازم شد (کاربر خواست Agent «به خاطر بسپاره») برو سراغ pgvector.

### ۳.۸ صف اجرا (Queue) برای کارهای طولانی

Agentها ممکنه چند دقیقه طول بکشن (چند بار جستجو، چند بار اجرای ابزار). این را نمی‌شه در یک HTTP request معمولی نگه داشت.

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Redis + BullMQ** | ساده، سریع برای راه‌اندازی، اکوسیستم Node.js خوب | خودت باید retry/observability رو دستی مدیریت کنی |
| **Temporal** | مدیریت workflowهای طولانی و چندمرحله‌ای با تضمین اجرای صحیح حتی بعد از کرش | یادگیری مفاهیمش (Workflow/Activity) وقت می‌بره؛ زیرساخت اضافه |
| **Celery (Python)** | استاندارد و باتجربه در اکوسیستم پایتون | نیاز به broker (Redis/RabbitMQ) جدا |

**توصیه:** شروع با **Redis + BullMQ** (یا معادل پایتونی‌اش، Celery) کافیه؛ Temporal رو وقتی لازم داری که workflowهای واقعاً پیچیده و حیاتی (مثل تراکنش مالی) داشته باشی در نظر بگیر.

### ۳.۹ مشاهده‌پذیری (Observability) و لاگ

| تکنولوژی | مزیت | عیب |
|---|---|---|
| **Langfuse (self-hosted یا cloud)** | مخصوص ردیابی «فکر کردن» Agent (هر tool call، هر prompt) | یک سرویس اضافه برای راه‌اندازی |
| **OpenTelemetry + Grafana** | استاندارد عمومی، قابل استفاده برای کل سیستم نه فقط AI | نیاز به تنظیم دستی dashboardها برای مفاهیم خاص Agent |

**توصیه:** از روز اول حداقل لاگ ساختاریافته (هر step از اجرای Agent: کدام tool، چه ورودی، چه خروجی، چند توکن) رو در دیتابیس خودت ذخیره کن؛ بعداً Langfuse رو روش اضافه کن.

### ۳.۱۰ محدودیت نرخ و صورتحساب (Rate limiting & Billing)

- شمارش توکن مصرفی هر اجرا (از پاسخ API این عدد رو می‌گیری) و ذخیره‌ی آن به ازای هر کاربر/Agent.
- محدودیت تعداد گام‌های حلقه‌ی اجرا (مثلاً حداکثر ۱۰ فراخوانی ابزار در هر اجرا) تا یک Agent در حلقه‌ی بی‌نهایت هزینه تولید نکنه.
- Stripe (metered billing) برای تبدیل مصرف توکن به صورتحساب، اگر قرار است محصول پولی باشه.

## ۴. جدول تصمیم سریع: کدام مسیر برای شروع؟

| لایه | انتخاب پیشنهادی برای MVP |
|---|---|
| Frontend | Next.js + WebSocket برای استریم چت |
| Auth + DB | Supabase (Postgres + Auth یک‌جا) |
| Backend Orchestrator | Python + FastAPI با حلقه‌ی دست‌ساز روی Claude tool-use |
| مهارت‌ها | فقط چند تابع از پیش‌نوشته (وب‌سرچ، HTTP محدود)، بدون اجرای کد دلخواه |
| صف اجرا | Redis + Celery (یا BullMQ اگر Node) |
| لاگ | جدول ساده‌ی Postgres برای هر step، بدون سرویس اضافه فعلاً |

## ۵. نقشه راه پیشنهادی

**فاز ۱ — MVP (اثبات مفهوم):**
Auth ساده + فرم ساخت Agent (system prompt + انتخاب چند tool از لیست ثابت) + یک حلقه‌ی اجرای واحد روی Claude API + محدودیت گام اجرا. بدون sandbox پیچیده، بدون حافظه، بدون چند-Agent.

**فاز ۲ — گسترش قابلیت:**
اضافه‌کردن صف اجرا برای کارهای طولانی، حافظه/RAG با pgvector، امکان اجرای کد در sandbox واقعی (Firecracker/gVisor)، داشبورد مشاهده‌پذیری.

**فاز ۳ — مقیاس و تیم‌سازی Agentها:**
چند Agent که با هم هماهنگ می‌شن (multi-agent orchestration مثل CrewAI/LangGraph)، مارکت‌پلیس مهارت‌های عمومی، billing کامل بر اساس مصرف.

## ۶. نکات امنیتی حیاتی (پیش از هر چیز دیگه بخون)

1. **هر مهارت باید allowlist باشه**، نه اینکه کاربر بتونه هر تابعی دلخواه تعریف کنه بدون بازبینی.
2. **خروجی ابزارها را کورکورانه اعتماد نکن** — یک صفحه‌ی وب یا نتیجه‌ی جستجو می‌تونه حاوی متنی باشه که تلاش می‌کنه به مدل دستور بده (Prompt Injection)؛ خروجی ابزار را به‌عنوان «داده»، نه «دستور معتبر»، به مدل بده و مرزهای واضح در system prompt تعریف کن.
3. **هر اجرا باید محدودیت منابع داشته باشه**: حداکثر زمان، حداکثر تعداد گام، حداکثر هزینه.
4. **کلیدهای API و اطلاعات حساس هرگز مستقیم در اختیار Agent قرار نگیره** — Agent باید از طریق یک لایه‌ی واسط به سرویس‌های خارجی دسترسی داشته باشه، نه با کلید خام در context.
5. **ایزوله‌سازی داده‌ی هر کاربر (tenant isolation)** در سطح دیتابیس و در سطح اجرای sandbox، تا Agent یک کاربر نتونه به داده‌ی کاربر دیگه دسترسی پیدا کنه.

---

اگه آماده‌ای، قدم بعدی طبیعی اینه که با **فاز ۱** شروع کنیم: یک اسکلت پروژه (Next.js frontend + FastAPI backend + Postgres) با یک Agent ساده که فقط یک مهارت («جستجوی وب») داره، بسازیم و واقعاً اجراش کنیم.
