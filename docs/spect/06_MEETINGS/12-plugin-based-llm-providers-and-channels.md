## ۲. بحث و تصمیمات

### موضوع ۱: جزئیات MKT-028 — مشارکت داوطلبانه‌ی تست

- **تصمیم:** کاربرانی که در سیستم اعلام می‌کنن مایل به تست پلاگین‌های جدیدن، پلاگین‌های در انتظار تست براشون نمایش داده می‌شه؛ می‌تونن نصب/استفاده/نظردهی کنن. **رایگان** براشون (هیچ هزینه‌ای دریافت نمی‌شه). اگه باگ گزارش کنن، **جایزه** می‌گیرن.
- این جزئیات کافیه برای شروع تسک‌بندی MKT-028 — دیگه «باز» نیست.

### موضوع ۲: روش‌های شارژ کیف‌پول

- **تصمیم:** کیف‌پول باید چند روش شارژ رو پشتیبانی کنه:
  1. **خودکار** — از طریق درگاه/پرداخت بانکی (automation).
  2. **دستی** — همون چیزی که الان به‌عنوان MVP-lite هست (admin manual credit).
  3. **کریپتو/استیبل‌کوین** — پرداخت با رمزارز.
- **بررسی پس‌زمینه:** این گسترش همون CP-013 (Billing واقعی، در `04_control-plane-backlog.md` به‌عنوان Later علامت‌گذاری شده بود) است — این جلسه فقط روش‌های پرداخت مشخص رو دقیق کرد، تصمیم اصلی («billing واقعی باید ساخته بشه») از قبل بود.

### موضوع ۳: بخش افیلیت در Platform

- **تصمیم:** Platform باید یک سیستم افیلیت (معرف/کد ارجاع، پورسانت روی مصرف/خرید کاربر معرفی‌شده) داشته باشه. جزئیات دقیق (درصد پورسانت، سطح‌بندی، نحوه‌ی پرداخت پورسانت) موکول به یک دور Q&A جدا قبل از پیاده‌سازی — این جلسه فقط اصل نیاز رو ثبت کرد.

### موضوع ۴: اتصال LLM Provider به‌عنوان Plugin (تغییر معماری بزرگ)

- **بررسی وضعیت فعلی (قبل از بحث):**
  - `env.ts`: `LLM_PROVIDER: z.enum(['anthropic', 'openai', 'deepseek', 'ollama'])` — مستقیم، بدون پلاگین.
  - `llm_configs` (`provider-config-service.ts`): سطح **Organization** — یک ردیف به‌ازای هر org، نه هر Agent.
  - مدل Agent اصلاً فیلد provider نداره.
  - پلاگین موجود (RT-079، `manifest.provider: {type: 'http', baseUrl}`) یک آداپتور HTTP عمومی برای *ابزار*هاست، نه برای LLM completion API — قرارداد فنی متفاوتیه.
- **ایده‌ی کاربر:** اتصال به LLM provider هم در Runtime هم در Platform باید یک **Plugin نصب‌شدنی** باشه. با نصب یک پلاگین provider، Runtime می‌تونه به اون provider وصل بشه؛ همون پلاگین در Platform هم برای AI Gateway استفاده می‌شه. کاربر Runtime بعد از نصب Runtime پیش‌فرض فقط Ollama داره (local یا cloud)؛ برای provider دیگه باید پلاگین (رایگان یا پولی) نصب کنه و بعدش با توکن خودش استفاده کنه. حتی **AI Gateway خودِ on4net هم یک پلاگین نصب‌شدنی می‌شه** (نه یک قابلیت built-in جدا).
- **مزیت این مدل (طبق توضیح کاربر):** دو مسیر مجزا برای هر Agent — (۱) نصب پلاگین AI Gateway → مدیریت هوشمند چندمدلی (مثل قبل)، (۲) نصب یک یا چند پلاگین Provider خاص و اختصاص به یک Agent → اون Agent فقط از همون provider خاص استفاده می‌کنه.
- **تحلیل Claude (قبل از تصمیم‌گیری نهایی):**
  - این تناقضی با «BYOK: هر Agent فقط یک provider فعال» نداره — این قاعده هنوز درسته، فقط واحدش از **کل Runtime instance** به **هر Agent جدا** منتقل می‌شه؛ یعنی حالا دو Agent مختلف توی یک Runtime instance می‌تونن هم‌زمان provider متفاوت داشته باشن (چیزی که امروز اصلاً ممکن نیست چون `llm_configs` سطح org است).
  - این یک تغییر معماری واقعی و بزرگه، نه یه patch کوچیک — نیاز به: فیلد provider-plugin روی Agent، مهاجرت سه provider موجود (anthropic/openai/deepseek) از env به پلاگین، و یک قرارداد فنی جدید برای «پلاگین LLM provider» (متفاوت از پلاگین HTTP-tool موجود RT-079).
- **تصمیم نهایی کاربر (بعد از تأیید Claude):**
  1. **بله، هرسه provider موجود (anthropic/openai/deepseek) به پلاگین مهاجرت کنن.** فقط Ollama (local یا cloud) به‌صورت مستقیم/بدون پلاگین باقی می‌مونه.
  2. **بله، سطح انتساب از Organization به per-Agent منتقل بشه.** هر Agent جدا می‌تونه provider-plugin خودش رو داشته باشه.

### موضوع ۵: کانال‌های ارتباطی (Telegram/WhatsApp) به‌عنوان Plugin + امنیت Device Approval

- **بررسی وضعیت فعلی:** `telegram-connector.ts` امروز فقط **outbound** است (فقط پیام می‌فرسته، `chat_id` مشخص). هیچ inbound handler، هیچ allowlist/approval‌ای وجود نداره. یعنی این یک قابلیت عمدتاً **جدید**ه، نه گسترش چیزی که از قبل نیم‌کاره بود.
- **تصمیم:** همه‌ی کانال‌های ارتباطی (Telegram، WhatsApp، …) باید Plugin باشن، نه کد hardcoded داخل Runtime.
- **الزام امنیتی ۱ — Device Approval:** وقتی یک کاربر خارجی (مثلاً یک فرستنده‌ی تلگرام) برای اولین‌بار به بات متصل‌شده پیام می‌ده، قبل از اینکه سیستم بهش جواب بده، اون «device» (منظور: هویت فرستنده‌ی خارجی — chat_id/شماره واتزآپ) باید توسط ادمین سازمان **approve** بشه.
- **الزام امنیتی ۲ — Device↔Agent binding:** بعد از approve شدن، باید مشخص بشه اون device دقیقاً به کدوم Agent (یا کدوم Agentها) اجازه‌ی باز کردن session/گفتگو داره — یک رابطه‌ی many-to-many بین device تأییدشده و Agentهای مجاز.

## ۴. اصلاحات کاربر (همون روز، بعد از بازخوانی خلاصه Claude)

- **CP-050 (افیلیت):** درصد پورسانت باید از طریق یک **فرم در پنل ادمین Platform** توسط خودِ ادمین تعیین بشه (نه یک عدد ثابت hardcoded). سطح‌بندی و روش پرداخت پورسانت هنوز باز مونده.
- **اصلاح مهم روی «device» (موضوع ۵):** تفسیر اولیه‌ی Claude («device = هویت فرستنده‌ی خارجی») ناقص بود. تعریف درست کاربر:
  - **device اصلی یعنی کلاینت وب/اپلیکیشن** — وقتی کاربر از طریق مرورگر یا (در آینده) اپ به Runtime وصل می‌شه. برای این حالت، تأیید **اختیاری/قابل‌تنظیم**ه: یک تنظیم توی Runtime، پیش‌فرض **خاموش** (تأیید لازم نیست)؛ اگه ادمین سیستم روشنش کنه، از اون به بعد device/session جدید قبل از اتصال حتماً باید تأیید بشه. → **RT-117 (تسک جدید)**.
  - **برای سیستم‌های خارجی مثل Telegram** (channel plugins، موضوع اصلی که RT-115 قبلاً بهش اشاره داشت)، تأیید **همیشه اجباریه، بدون تنظیم/خاموش‌کردن** — این بخش از تفسیر اولیه‌ی Claude درست بود، فقط باید به‌عنوان یک مورد جدا از device وب/اپ دیده بشه، نه زیرمجموعه‌ی همون یک مفهوم.
  - نتیجه: RT-115 به «تأیید اجباری کانال خارجی» محدود شد (بدون تغییر جوهری)، و یک تسک کاملاً جدید (**RT-117**) برای تأیید قابل‌تنظیم device وب/اپ اضافه شد.
- **مورد ۳ (رابطه‌ی پلاگین AI Gateway با توکن RT-101) تأیید شد:** این دو یکی هستن — «نصب پلاگین AI Gateway» دقیقاً همون «تنظیم توکن AI Gateway» جلسه ۸ است، نه دو سازوکار جدا. CP-051 بالا اصلاح شد.
