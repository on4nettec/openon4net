## ۲. بحث و تصمیمات

### موضوع ۱: Multi-model/multi-provider routing داخل Runtime

- **بحث:** کاربر پرسید آیا می‌شه داخل Runtime (self-host) بین چند مدل محلی Ollama ترکیبی استفاده کرد، یا بین Ollama و یک سرویس معروف مثل OpenAI سوییچ کرد تا هزینه کم بشه؛ و استدلال کرد که چون اینجا از یک سرویس آنلاین با کلید خودِ سازمان استفاده می‌شه (نه سرویس مدیریت‌شده‌ی O2N)، شاید نقض مرز AI Gateway نباشه.
- **یافته (بررسی مستندات/کد قبل از پاسخ):** این دقیقاً همون چیزیه که در تصمیم معماری موجود عمداً از Runtime بیرون گذاشته شده — نه یک‌بار، بلکه دو جای مستقل مستند شده:
  - کد، `gateway/src/services/llm-service.ts`: _«BYOK: exactly one provider, one retry, no cross-provider fallback... Runtime deliberately does not reimplement [routing/fallback-chain/circuit-breaker]»_
  - کد، `gateway/src/services/provider-config-service.ts`: _«still no cross-provider routing/fallback (that's Control Plane's job)»_
  - سند، `docs/spect/02_ARCHITECTURE/14-monorepo-layout.md` خط ۸۴: _«"AI Gateway" (روتینگ بین چند LLM) فقط داخل openon4net-control-plane/gateway پیاده می‌شود؛ openon4net-runtime/gateway صرفاً یک کانکتور ساده‌ی مستقیم به یک LLM (BYOK) دارد.»_ و خط ۱۴۵: _«Managed AI Gateway: تنها جایی که routing/fallback/cost-tracking بین چند LLM پیاده می‌شود.»_
- **تناقض کشف‌شده:** یک سند قدیمی‌تر (`docs/spect/02_ARCHITECTURE/02-ai-gateway.md` §۱.۱) می‌گه AI Gateway یک حالت self-host هم داره که **داخل خودِ Runtime** اجرا می‌شه — دقیقاً همون استدلالی که کاربر مطرح کرد. یعنی بین دو سند خودمون ناهم‌خوانی واقعی هست؛ سند جدیدتر (`14-monorepo-layout.md`) و کد واقعی هر دو طرف محدودتر (فقط Control-Plane) رو گرفتن.
- **تحلیل Claude:** این محدودیت به‌نظر یک مرز کسب‌وکاری هست، نه صرفاً فنی. از نظر فنی پیاده‌سازیش خیلی ساده‌ست (یک لایه‌ی انتخاب model/provider قبل از `LlmService`). ولی اگه Runtime رایگان بتونه بین چند مدل — حتی کاملاً BYOK و محلی — هوشمند روتینگ کنه، دیگه انگیزه‌ی خرید Control-Plane مدیریت‌شده (تنها محل مستندشده‌ی routing/failover/cost-tracking چندمدلی) از بین می‌ره. این احتمالاً همون دلیل نانوشته‌ای هست که باعث شده این قابلیت عمداً از Runtime حذف بشه.
- **اصلاح کاربر (بعد از بحث اول):** کاربر یک راه‌حل میانه پیشنهاد داد که خط قرمز مستندات فعلی رو نگه می‌داره ولی محدودیت رو منطقی‌تر می‌کنه:
  - **در Runtime:** کاربر می‌تونه دقیقاً **یک سرویس خارجی BYOK** (مثل OpenAI/Anthropic/DeepSeek) تنظیم کنه، **و هم‌زمان از Ollama محلی خودش هم استفاده کنه**. یعنی نه «تک‌provider محض» فعلی، بلکه «تک-provider خارجی + Ollama محلی» — یک gateway کوچیک داخل Runtime باید بین این دو (یک سرویس ابری + Ollama) مدیریت کنه، نه بین چند سرویس ابری.
  - **در Control-Plane (AI Gateway مدیریت‌شده):** اونجا کاربر می‌تونه **هم‌زمان از چند مدل/سرویس ابری مختلف** استفاده کنه — **به‌همراه همون Ollama محلی خودش که روی سیستمش داره**. یعنی ترکیب «N سرویس ابری از طریق Control-Plane + Ollama محلی از طریق Runtime» یک قابلیت **منحصربه‌فرد** می‌شه که نه Runtime تنها می‌تونه بدش (چون محدود به یک سرویس خارجیه)، نه یک رقیب بیرونی می‌تونه راحت کپی کنه (چون به Ollama محلی کاربر دسترسی نداره).
- **تحلیل تکمیلی Claude:** این مدل، انگیزه‌ی خرید Control-Plane رو به‌جای «یک قابلیت که کاملاً حذف شده»، به «یک قابلیت واقعاً برتر و غیرقابل‌تکرار» تبدیل می‌کنه — که از نظر محصول قوی‌تره. یک سؤال فنی باز می‌مونه: برای اینکه Control-Plane بتونه یک درخواست رو به Ollama محلی کاربر بفرسته، باید یک مسیر ارتباطی برعکس (Control-Plane → دستور اجرا → Runtime محلی) وجود داشته باشه؛ الان فقط یک جهت (`Runtime → Control-Plane` برای check-in، طبق T-CP-007) پیاده‌سازی شده. این باید در فاز پیاده‌سازی طراحی بشه (مثلاً webhook/polling از Runtime برای گرفتن دستورهای routing).
- **قید مهم کاربر (تأکید صریح):** هر جا «Ollama» گفته می‌شه، منظور فقط **Ollama لوکال** است — یعنی روی همون سیستم/شبکه‌ی داخلی کاربر، **نه یک Ollama ابری/میزبانی‌شده روی اینترنت**. IP مقصد باید لوکال باشه (مثل `localhost`/`127.0.0.1` یا رنج IP خصوصی شبکه‌ی داخلی)، نه یک IP عمومی اینترنتی. این قید باید در پیاده‌سازی enforce بشه (اعتبارسنجی آدرس Ollama قبل از اتصال، رد کردن IPهای عمومی) — دقیقاً برعکسِ محافظت SSRF موجود در `webhook-connector.ts` که آدرس‌های لوکال/خصوصی رو **رد** می‌کنه؛ اینجا باید آدرس‌های **غیر-لوکال** برای فیلد Ollama رد بشن.
- **تصمیم:** ⏳ **جهت کلی تأیید شد توسط کاربر، جزئیات هنوز باز:**
  1. Runtime: یک BYOK خارجی + Ollama لوکال (فقط IP داخلی/private، نه اینترنت)، با یک gateway ساده‌ی داخلی بین این دو (نه بین چند سرویس ابری).
  2. Control-Plane: چند سرویس ابری هم‌زمان + همون Ollama لوکال کاربر (دسترسی متقابل)، به‌عنوان تمایز اصلی نسخه‌ی مدیریت‌شده.
  - جزئیات فنی (مسیر ارتباطی Control-Plane ↔ Ollama لوکال، معیار انتخاب مدل در هر دو حالت، enforce کردن قید IP لوکال) موکول به فاز پیاده‌سازی شد.

### موضوع ۲: Fine-tuning برای سازمان‌های بزرگ

- **بحث:** کاربر قبلاً به این نتیجه رسیده بود که fine-tuning باید برای سازمان‌های بزرگ در سیستم گنجانده بشه، ولی هیچ ایده‌ای از شکلش نداشت. یک دور Q&A برای scope کردنش انجام شد.

- **پیشنهاد معماری Claude (برای تصویب در جلسه‌ی بعد، نه یک تصمیم قطعی):**
  ایده‌ی خودِ کاربر (self-host + فروش لایسنسی + فقط از Marketplace) با معماری موجود O₂N جور درمیاد، بدون نیاز به یک مکانیزم کاملاً جدید:
  1. **اجرا همیشه self-host می‌مونه** (روی زیرساخت/GPU خودِ سازمان) — سازگار با اصل موجود پروژه که داده‌ی حساس هیچ‌وقت از Runtime خارج نمی‌شه.
  2. **قفل‌شدن قابلیت پشت یک license/feature-flag** — با استفاده از همون مکانیزم activation موجود Control-Plane (`featureFlags` در پاسخ `POST /activation/check-in`، دقیقاً همون چیزی که T-CP-007 امسال بهش وصل شد؛ ببینید `docs/spect/DONE.md`). یعنی از زیرساخت موجود استفاده می‌شه، نه یک سیستم لایسنسینگ کاملاً جدا.
  3. **فروش از طریق Marketplace (Plane 4)** — دقیقاً مثل یک Plugin پولی، با همون pipeline نصب/قیمت/entitlement که امسال ساخته شد (`docs/spect/02_ARCHITECTURE/12-marketplace-service.md`)؛ فقط این «Plugin» به‌جای کد اجراشونده، یک feature-unlock هست.
  4. این یعنی fine-tuning **لزوماً نیاز به «پروژه‌ی کاملاً مجزا» در سطح ریپو نداره** — می‌تونه یک ماژول/سرویس داخل Runtime (یا یک companion service که Runtime باهاش صحبت می‌کنه) باشه که Marketplace فقط قفلش رو باز می‌کنه. اگه کاربر ترجیح بده جدا نگهش داره (مثلاً به‌خاطر پیچیدگی وابستگی‌های ML که نباید Runtime اصلی رو سنگین کنه)، اون هم قابل قبوله — این یک تصمیم بعدی و جدا از مدل درآمدیه.
- **⚠️ یادآوری صادقانه‌ی مقیاس:** پیاده‌سازی واقعی fine-tuning (GPU orchestration، یک training framework مثل LoRA/PEFT/axolotl، ارزیابی مدل، model versioning، و سرو کردن نتیجه از طریق Ollama) یک حوزه‌ی مهندسی کاملاً جدا و بزرگ‌تر از هر چیزیه که تا الان توی این پروژه ساخته شده — فازهای ۰ تا ۳ همه CRUD/API/orchestration بودن، نه ML training واقعی. این باید به‌عنوان یک فاز مستقل و واقعاً بزرگ دیده بشه، نه یک batch مثل batchهای قبلی این جلسات.
- **تصمیم:** ⏳ **باز — نیاز به تصویب کاربر روی پیشنهاد معماری بالا، قبل از هرگونه شروع پیاده‌سازی.**

### موضوع ۳: پلن‌های اشتراکی Control-Plane برای هاست‌ها/reseller (کاربرانی که سخت‌افزار مناسب ندارن)

- **بحث:** کاربر یک دسته‌ی جدید از کاربران رو مطرح کرد: کسانی که سخت‌افزار مناسب برای self-host ندارن. برای این‌ها دو مسیر هست:
  1. کمک به self-host روی زیرساخت ضعیف‌تر خودشون (قبلاً تا حدی پوشش داده شده — self-host پیش‌فرض MVP هست).
  2. قرارداد با شرکت‌های **هاست‌کننده** که مسئولیت راه‌اندازی/میزبانی Runtime رو برای کاربر نهایی به عهده می‌گیرن (کاربر به یک مثال مشابه «openclaw» اشاره کرد — شرکت‌های هاست که این سرویس رو به مشتری‌های خودشون می‌دن).
  - در حالت دوم، چون این هاست‌ها ممکنه ده‌ها/صدها نمونه‌ی Runtime برای مشتری‌های مختلف بالا بیارن، مصرف AI Gateway Control-Plane (اگه مشتری‌ها از اون استفاده کنن) به‌شدت بالا می‌ره.
  - برای اینکه خودِ هاست‌ها هم انگیزه‌ی مالی داشته باشن این سرویس رو بفروشن (نه فقط O2N سود کنه)، کاربر پیشنهاد داد یک **پلن اشتراکی مخصوص هاست/reseller** برای AI Gateway Control-Plane تعریف بشه — با یک سقفِ **تعداد دامنه/نمونه‌ی متصل** (مثلاً یک پلن «۱۰۰» که تا ۱۰۰ دامنه/Runtime instance رو پوشش می‌ده؛ بعد از عبور از سقف، محدودیت اعمال می‌شه).
  - این نیاز به یک **API مدیریتی** داره تا خودِ هاست بتونه instanceهای متصل به سهمیه‌ش رو مدیریت کنه — این به‌عنوان کار «فاز توسعه» در نظر گرفته شده، نه چیزی برای الان.
- **تحلیل Claude:** این یک نقش جدید (Reseller/Host) هست که امروز توی مدل داده‌ی Control-Plane وجود نداره — الان فقط «Organization» با یک activation key هست (`POST /admin/activation-keys`، فقط سمت ادمین خودِ O2N). یک لایه‌ی جدید لازمه:
  - یک entity «Reseller/Host account» جدا از Organization، با یک پلن سهمیه‌دار (مثلاً `max_connected_instances`).
  - یک API scope-شده (نه ادمین کامل O2N) که خودِ هاست بتونه در سقف سهمیه‌ش activation key صادر/لغو کنه برای مشتری‌های خودش.
  - این از نظر مفهومی شبیه مدل‌های reseller/white-label رایج در صنعت هاستینگ هست (مثل reseller hosting cPanel/WHM)، نه چیز بی‌سابقه‌ای.

- **توضیح ارزش‌آفرینی کاربر (چرا این مدل برای شرکت‌های کوچیک جذابه):** یک شرکت کوچیک به‌جای خرید مستقیم اشتراک AI Gateway Control-Plane برای خودش، می‌تونه از طریق هاست به قیمت **ارزون‌تر** به همون قابلیت دسترسی پیدا کنه — چون هاست از قبل پلن reseller رو عمده خریده، و کسب‌وکار اصلی هاست «فروش هاستینگ» است نه «فروش AI Gateway»، پس می‌تونه AI Gateway رو به‌عنوان یک مزیت رقابتی ارزون/جذاب به مشتری‌هاش بده تا اونا رو به سمت هاستینگ خودش جذب کنه.
- **ایده‌ی جدید — Award/تشویق هاست‌ها:** در فاز طراحی وب‌سایت رسمی خودِ on4net، یک award/معرفی برای هاست‌هایی که بیشترین استفاده رو از AI Gateway داشتن در نظر گرفته بشه — یک مکانیزم بازاریابی/تشویقی (نه یک مکانیزم درآمدی مستقیم)، مشابه برنامه‌های partner/badge رایج (مثلاً AWS/Cloudflare Partner Badges).
- **تصمیم:** ⏳ **جهت‌گیری کلی و مدل قیمت‌گذاری/مسئولیت تأیید شد. جزئیات باز مونده برای فاز پیاده‌سازی:** نسبت دقیق قیمت ثابت به سهم درآمد، شکل دقیق سهمیه‌بندی/API، و معیار award (مثلاً بر اساس تعداد instance فعال یا حجم مصرف واقعی).
