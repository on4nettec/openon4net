## ۱. دستور کار (Agenda)

کاربر بدون Q&A اولیه، یک‌جا یک ویژن کامل برای AI Gateway مطرح کرد (شبیه OpenRouter.ai)، و خواست همین جلسه ثبت و تسک‌بندی بشه. Claude قبل از تسک‌بندی، وضعیت فعلی کد رو بررسی کرد (نه فرض) و چند سؤال معماری واقعاً باز رو با کاربر مطرح کرد.

## ۲. وضعیت فعلی (بررسی شده قبل از بحث، نه فرض)

- **CP-012 موجود** (`ai-gateway-service.ts`) کاملاً **org-level** است — هیچ مفهوم Workspace/Token در Platform وجود نداره. یک زنجیره‌ی provider مرتب‌شده (priority) به‌ازای هر Organization، با fallback ساده (اولین موفق برنده است)، فقط ۴ provider (`anthropic`/`openai`/`deepseek`/`ollama` — همون `SupportedProvider` در `@o2n/llm-providers`)، هزینه فقط در سطح org ثبت می‌شه (نه workspace/token/agent).
- **Platform اصلاً entity «Workspace» ندارد** — فقط `Organization`. (برخلاف Runtime که workspace داره.)
- سند آرمانی `02_ARCHITECTURE/02-ai-gateway.md` (v1.0) از قبل مفاهیم مشابهی (Model Router، Cost Management، Failover، Circuit Breaker، Cache) رو در حد ویژن مستند کرده بود، ولی CP-012 عمداً فقط زیرمجموعه‌ی واقعی backlog (`04_control-plane-backlog.md`'s US-630..633) رو ساخت، نه کل سند آرمانی. این جلسه اون سند آرمانی رو با جزئیات مشخص‌تر (workspace/token) تکمیل/عملیاتی می‌کنه.
- **نکته‌ی فنی مهم Claude (قبل از تصمیم‌گیری روی تعداد provider):** «۴۰۰ provider» به‌معنی ۴۰۰ آداپتور کد جدا نیست — دقیقاً همون‌طوری که OpenRouter خودش هم این‌کار رو نمی‌کنه. اکثریت قریب‌به‌اتفاق این providerها (Groq, Together AI, Fireworks, Mistral, Perplexity, DeepInfra, Azure OpenAI, ...) از همون پروتکل سیمی OpenAI-compatible استفاده می‌کنن که Runtime از قبل آداپتورش رو داره. مسیر واقع‌بینانه: یک **کاتالوگ provider** (داده، نه کد) — base URL + مدل‌های پیش‌فرض + قیمت + دسته‌بندی — روی فقط ۲-۳ آداپتور پروتکل واقعی (OpenAI-compatible که اکثریت رو پوشش می‌ده + Anthropic-native که از قبل داریم). کاربر این تحلیل رو پذیرفت.

## ۳. ویژن کامل کاربر (نقل تقریبی، فارسی اصلی حفظ شد)

- کاربر می‌تونه یک **Workspace** تعریف کنه و یک یا چند **API Key/توکن** بهش اختصاص بده.
- هر توکن می‌تونه محدود بشه: از نظر **پول** (سقف مصرف)، از نظر **کدام مدل‌ها**، از نظر **کدام providerها** (دسترسی یک workspace محدود به یک یا چند provider خاص)، از نظر **زمان استفاده**.
- کاربر می‌تونه **ترتیب استفاده از providerها** رو خودش مشخص کنه؛ providerها **دسته‌بندی** دارن.
- نزدیک به ۴۰۰ provider مثل OpenRouter — تمایز اصلی O2N نسبت به OpenRouter: **انتخاب هوشمند provider** (نه فقط لیست بی‌ترتیب) + محدودسازی دسترسی هر workspace به provider(های) خاص.
- **در Runtime:** اگه یک Agent از AI Gateway استفاده کنه، کاربر برای همون Agent یک توکن درخواست/استفاده می‌کنه.
- **در Platform:** داشبورد باید میزان مصرف کل و مصرف هر workspace رو نشون بده؛ workspace بتونه به Google Drive وصل بشه (بعدها درایورهای آنلاین دیگه هم اضافه می‌شن)؛ هزینه‌ی مصرف هر provider در داشبورد مدیریت اصلی دیده بشه (هر کاربر چقدر از کدام provider مصرف کرده)؛ در **landing page** یک نمای میزان مصرف/محبوبیت providerها (پرمصرف‌ترین provider) نمایش داده بشه.
- **در Runtime چت:** وقتی از AI Gateway استفاده می‌شه، مشخص باشه هر پیام به کدام provider ارسال شده — تا بعداً مدیر Runtime بفهمه بیشترین مصرف هر Agent از کدام provider بوده.

## ۵. جزئیاتی که عمداً باز ماندند (موکول به فاز پیاده‌سازی، نه فراموش‌شده)

- **معنای دقیق «محدودیت زمانی توکن»**: بازه‌ی اعتبار (`validFrom`/`validUntil`) یا محدودیت ساعت/روز هفته (شبیه الگوی cron که در RT-088 برای Agent Schedule ساختیم)؟ هر دو ممکنه، تصمیم در فاز پیاده‌سازی.
- **فرمول دقیق امتیازدهی «انتخاب هوشمند»**: وزن نسبی هزینه/در‌دسترس‌بودن/latency، و اینکه circuit-breaker (مثل سند آرمانی `02-ai-gateway.md`) واقعاً ساخته بشه یا یک نسخه‌ی ساده‌تر (شمارش خطای اخیر) کافیه.
- **لیست دقیق و نهایی provider‌های فاز اول** (۱۰-۱۵ تا) و مدل‌های هرکدوم — Claude این جلسه فقط یک پیشنهاد اولیه داد؛ تثبیت نهایی قبل از شروع پیاده‌سازی provider catalog.
- **دسته‌بندی providerها**: محورهای دسته‌بندی (مثلاً سرعت/هزینه/reasoning/vision) هنوز مشخص نشده.
- **اتصال Google Drive به Workspace**: این عملاً یک قابلیت جدا از AI Gateway است (ذخیره‌سازی/بک‌آپ)، فقط از طریق همون entity جدید Workspace به این جلسه وصل شد. باید از اول به‌شکل یک انتزاع «storage connector» طراحی بشه (نه Google-specific) چون کاربر صریحاً گفت درایورهای دیگه بعداً اضافه می‌شن.

## ۶. افزوده (همون روز) — کاتالوگ فنی مدل، کمیسیون، Model Fusion، توکن پیش‌فرض/به‌ازای Agent

کاربر بعد از ثبت اولیه، چند نکته‌ی تکمیلی اضافه کرد:

### ۶.۱ کاتالوگ فنی مدل — فیلدهای دقیق

CP-037 (کاتالوگ Provider) با این فیلدهای مشخص گسترش پیدا می‌کنه — به‌ازای هر (Provider, Model):

`Provider`, `Model`, `Parameters`, `Training Tokens`, `Context Window`, `Architecture`,
`Active Parameters` (برای MoE), `Benchmark Scores`, `Reasoning Ability`, `Inference Speed`,
`Latency`, `Knowledge Cutoff`, `Multimodal Support`, `Tool Calling / Function Calling`,
`Fine-tuning Support`, `Memory Support`, `License`, `Hardware Requirement`, `Price / M tokens`
(ورودی/خروجی جدا).

قیمت باید «هر چند وقت یک‌بار» به‌روز بشه. **محدودیت واقعی، نه گپ فراموش‌شده:** هیچ API عمومی
یکپارچه‌ای که قیمت لحظه‌ای همه‌ی providerها رو بده وجود نداره (OpenRouter خودش هم این کار رو
با نگهداری دستی/نیمه‌دستی کاتالوگش انجام می‌ده). مسیر واقع‌بینانه برای فاز اول: فیلد قیمت
**قابل‌ویرایش دستی توسط ادمین Platform** + یک `updatedAt` روی هر ردیف کاتالوگ (نه scraping
خودکار از سایت هر provider — اون یک کار جدا و شکننده برای هر provider جداگانه‌ست، در آینده
قابل‌اضافه‌شدن provider-به-provider اگه لازم شد).

### ۶.۲ دسته‌بندی «این مدل برای چه کاریه» (الهام از openrouter.ai/models)

جدا از کاتالوگ فنی بالا، یک محور دسته‌بندی «کاربردی» هم اضافه می‌شه — شبیه چیزی که در
openrouter.ai/models هست (مثل «Roleplay»، «Programming»، «Marketing copy»، ...): هر مدل
می‌تونه به یک یا چند دسته‌ی کاربردی نسبت داده بشه، مستقل از دسته‌بندی فنی (سرعت/هزینه/...).

### ۶.۳ کمیسیون Platform روی مصرف

مدیر Platform باید بتونه یک **درصد کمیسیون** روی قیمت خام هر provider (به ازای هر یک میلیون
توکن) تنظیم کنه — یعنی چیزی که از workspace/توکن کسر می‌شه = قیمت خام provider + کمیسیون
Platform. این باید در همون تنظیمات قیمت‌گذاری کاتالوگ (۶.۱) قابل تنظیم باشه، نه هاردکد.

### ۶.۴ Model Fusion — فقط در وب، نه API

کاربر می‌تونه برای یک درخواست از **چند مدل هم‌زمان** استفاده کنه («Model Fusion») — ولی این
قابلیت **فقط در وب‌سایت Platform** در دسترسه (مثلاً یک playground/chat)، **نه از طریق API**ی
که Runtime باهاش صحبت می‌کنه. **جزئیات باز (نه تصمیم‌گرفته‌شده):** مکانیزم دقیق fusion
مشخص نشده — آیا پاسخ چند مدل کنار هم نمایش داده می‌شه (مقایسه‌ای)، یا یک مدل داور
(judge/synthesizer) پاسخ‌ها رو ترکیب می‌کنه؟ این باید قبل از پیاده‌سازی مشخص بشه.

### ۶.۵ توکن پیش‌فرض org-wide + override اختیاری به‌ازای هر Agent

مدیر Runtime می‌تونه یک **توکن اصلی/پیش‌فرض** برای کل سازمان تعریف کنه؛ اگه خواست، می‌تونه
علاوه بر اون، به‌ازای هر Agent یک توکن جداگانه هم تعریف کنه. Agentهایی که توکن اختصاصی
ندارن، از توکن پیش‌فرض سازمان استفاده می‌کنن. **این یعنی RT-101 (بخش J پایین) گسترش پیدا
کرد:** یک تنظیم org-level (توکن پیش‌فرض) + یک override اختیاری در سطح هر Agent، نه فقط
یک فیلد per-agent مستقل.

علاوه بر این، هر توکن (چه پیش‌فرض چه per-agent) باید یک **حالت قابل‌تنظیم** داشته باشه:
مستقیم از provider خودِ سازمان استفاده کنه (BYOK فعلی Runtime) یا از **AI Gateway
مدیریت‌شده‌ی Platform** — یعنی این دو مسیر (BYOK موجود Runtime و AI Gateway جدید Platform)
کنار هم می‌مونن و انتخابشون per-token قابل‌تنظیمه، نه اینکه یکی جایگزین اون یکی بشه.
