## ۲. توضیح مجدد: webhook relay (جلسه ۱۰) چیست

> کاربر جلسه ۱۰ رو نفهمیده بود؛ این خلاصه‌ی ساده‌شده‌ست (جزئیات کامل همون فایل جلسه ۱۰).

**مشکل:** خیلی از کاربرانی که Runtime رو روی سیستم شخصی/لوکال خودشون نصب می‌کنن (نه سرور با IP عمومی)، Runtime‌شون فقط از طریق `localhost` در دسترسه. سرویس‌هایی مثل Telegram Bot API یا Stripe که می‌خوان به Runtime «webhook» بفرستن (یعنی وقتی یه اتفاق افتاد، خودشون یه HTTP request به Runtime بزنن) نمی‌تونن به `localhost` کاربر برسن — چون `localhost` فقط روی خودِ سیستم کاربر معنی داره، نه از بیرون.

**راه‌حل:** چون Platform (نه Runtime) روی یه سرور با آدرس عمومی واقعی اجرا می‌شه:
۱. کاربر توی Platform یه «webhook relay» می‌سازه، یه اسم بهش می‌ده، Platform یه URL عمومی بهش می‌ده (مثلاً `https://platform.on4net.com/v1/webhook-relay/abc123`).
۲. کاربر همون URL رو به Telegram/Stripe/… می‌ده (نه آدرس لوکال خودش).
۳. Runtime کاربر یه اتصال باز (WebSocket) به Platform نگه می‌داره — این اتصال از سمت Runtime شروع می‌شه، پس مشکل `localhost`/فایروال نداره (مثل اینکه شما به یه سایت زنگ بزنید، نه اینکه اون سایت بتونه به شما زنگ بزنه).
۴. وقتی Telegram/Stripe به اون URL عمومی Platform سیگنال می‌فرسته، Platform همون سیگنال رو از طریق همون اتصال باز، به Runtime کاربر پاس می‌ده.
۵. Runtime دقیقاً همون کاری رو می‌کنه که اگه مستقیم بهش رسیده بود می‌کرد.

یعنی Platform اینجا فقط نقش «واسطه»/امانت‌دار رو داره — منطق واقعی (کدوم agent/workflow/tool اجرا بشه) کاملاً دست Runtime می‌مونه.

## ۴. بحث و تصمیمات معماری (مرور کامل کاربر)

### زمینه: معماری ۴ پروژه (نقل کاربر)

- **Runtime** (open-source، رایگان): دو بخش `gateway` (مغز اجرایی) + `web` (UI) — الان با پوشه از هم جدان، احتمالاً بعداً به دو ریپو جدا تبدیل می‌شن. در آینده اپ موبایل/دسکتاپ هم اضافه می‌شه.
- **Platform** (بخش درآمدی، حل مشکل provider/ابزار): از ۴ بخش تشکیل شده — UI (وب، بعداً اپ)، gateway (مرکز ارتباط + مسئول میانجی‌گری Runtime↔Memory/Marketplace)، **Memory**، **Marketplace**.
- **Memory**: کاربران با پرداخت به دو لایه حافظه‌ی سطح بالا دسترسی دارن؛ **fine-tuning هم مسئولیت Memory است** (نه یه پروژه جدا).
- **Marketplace**: مثل وردپرس — پلاگین‌های ساخته‌شده از ابزارها توسط برنامه‌نویسان آپلود/فروخته می‌شن، ولی قبلش تست می‌شن — یا از طریق افراد داوطلب تست، یا از طریق sandbox + AI agent (این دومی از قبل ساخته شده، MKT-025).
- **Memory و Marketplace هیچ UI مستقیم ندارن** — فقط از طریق Platform gateway/UI به کاربر سرویس می‌دن.

### موضوع ۱: ارتباط gateway↔web داخل Runtime

- **بررسی کد:** `web/lib/api-client.ts` (۹۷۵ خط) — ۱۱۴+ متد API، همه از طریق یک تابع `request()` که `fetch()` (REST) صدا می‌زنه. فقط چت (RT-090) از WebSocket استفاده می‌کنه.
- **تصمیم کاربر:** **هیبرید** — فقط بخش‌های زنده/بلادرنگ روی WebSocket برن، نه همه‌چیز. CRUD معمولی (ساخت Agent، لیست User، …) همون REST بمونه.
- **مصداق‌های «زنده/بلادرنگ» که باید WebSocket بشن (پیشنهاد Claude برای تسک‌بندی):** وضعیت اجرای workflow run (RT-105/subworkflow هم شاملش می‌شه)، وضعیت آنلاین/مشغول‌بودن Agent، نوتیفیکیشن تأیید (approval) بلادرنگ. جزئیات دقیق‌تر موکول به فاز پیاده‌سازی.

### موضوع ۲: مسیر مستقیم Runtime→Marketplace (بدون Platform)

- **بررسی کد:** `marketplace-client.ts`'s `marketplaceRequest()` — `useProxy = Boolean(securityToken && env.CONTROL_PLANE_URL)`؛ اگه یکی از این دو نباشه (مثلاً سازمان activate نشده)، مستقیم با `MARKETPLACE_API_KEY` ثابت به Marketplace وصل می‌شه (RT-093). این دقیقاً با معماری جدید («کاربران Runtime فقط از طریق Platform gateway به Marketplace/Memory دسترسی دارن») در تناقضه.
- **تصمیم کاربر:** **مسیر مستقیم حذف بشه.** از این به بعد، حتی مرور (discovery/browse) هم باید از طریق Platform بره — یعنی activation برای هرگونه دسترسی به Marketplace (نه فقط عملیات نیازمند auth مثل نصب/فروش) اجباری می‌شه.

### موضوع ۳: پروتکل ارتباطی Platform↔Memory/Marketplace + یک اصل کلی جدید

- **بررسی کد:** ارتباط فعلی Platform→Marketplace/Memory یک HTTP proxy معمولیه (`CP-032/034`)، نه WebSocket.
- **اصل کلی که کاربر مطرح کرد (مهم‌ترین تصمیم این جلسه):** **«هرجا ارتباط دوطرفه لازمه، WebSocket؛ وگرنه HTTP کافیه.»** این یک قاعده‌ی کلی پروژه‌ست، نه فقط جواب یه سؤال خاص.
- **اعمال این اصل روی Platform↔Memory/Marketplace (تحلیل Claude، طبق اصل بالا):**
  - عملیات ساده‌ی همزمان (submit/list/discover/install، query حافظه) — تک request/response‌ان، نیازی به push از سمت Memory/Marketplace به Platform ندارن → **HTTP همون‌جا کافیه، تغییری لازم نیست.**
  - **دو استثنای واقعی که به push/بلادرنگ نیاز دارن:**
    1. **Fine-tuning job status** (Memory) — یه job طولانی‌مدته؛ Platform/کاربر باید پیشرفتش رو بدون poll مداوم ببینه → **WebSocket لازمه.**
    2. **Sandbox test status** (Marketplace، روی MKT-025 موجود) — تست پلاگین هم می‌تونه طول بکشه؛ ناشر باید نتیجه رو بلادرنگ ببینه → **WebSocket لازمه.**
- این تحلیل («کجا واقعاً دوطرفه‌ست») پیشنهاد Claude‌ه بر اساس اصل کاربر، نه یه تصمیم مجزای دیگه از کاربر — اگه مصداق دیگه‌ای هم به ذهن‌تون رسید، باید اضافه بشه.

### موضوع ۴: مسئولیت Fine-tuning → Memory

- **تصمیم:** Fine-tuning رسماً مسئولیت **Memory** است (نه یک پروژه‌ی جدا که جلسه ۳ باهاش کلنجار می‌رفت، نه بخشی از Platform's AI Gateway). این جلسه۳'s سؤال باز رو تا همین حد جواب می‌ده؛ جزئیات فنی (فرمت داده، محل اجرای واقعی training) هنوز باز می‌مونه.
- **نیازمندی‌های فوری که باید همین الان لحاظ بشه (طبق درخواست کاربر):**
  - یک مدل **job** (نه synchronous) برای fine-tuning — چون training واقعاً زمان می‌بره.
  - وضعیت job از طریق WebSocket قابل مشاهده باشه (موضوع ۳ بالا).
  - منبع داده: هم استخراج خودکار از Memory Engine موجود، هم آپلود دستی دیتاست (طبق جلسه ۳).
  - محل واقعی اجرای training (GPU/سرویس بیرونی؟) هنوز باز است — جزو جلسه‌ی بعدی fine-tuning می‌مونه.

### موضوع ۵: مدل تست Marketplace — مشارکت انسانی جدید

- **وضعیت موجود:** MKT-025 (sandbox test-gate خودکار + AI agent) از قبل ساخته شده.
- **نیازمندی جدید:** یک مسیر موازی که در اون **افراد داوطلب** بتونن در تست پلاگین‌ها مشارکت کنن (قبل از انتشار عمومی) — جزئیات دقیق (چطور داوطلب می‌شن، امتیازدهی/پاداش، آستانه‌ی تعداد تأیید لازم) موکول به تسک‌بندی/پیاده‌سازی.
