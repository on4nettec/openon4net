# جلسه ۶ — ۲۰۲۶-۰۷-۱۷

**حضور:** کاربر (owner)، Claude
**مدت:** —
**موضوع:** بازبینی تعریف رسمی Agent + خلأهای شناسایی‌شده در Runtime (tool-calling واقعی، delegation بین Agentها، Skill builder، Schedule، LLM provider UX، WebSocket، i18n کامل، اولین اجرا/activation)

---

## ۱. دستور کار (Agenda)

این جلسه با یک سؤال درباره‌ی تعریف «Agent» شروع شد (چه در کد Runtime چه در تئوری عمومی
AI agents)، و به یک بازبینی کامل از فاصله‌ی بین تعریف رسمی (`00_VISION/06-glossary.md`، همان
جلسه گسترش داده شد) و پیاده‌سازی واقعی Runtime کشیده شد. کاربر یک لیست از observationها/
باگ‌ها/نیازها داد؛ هر مورد جداگانه در کد بررسی شد (نه صرفاً بر اساس گفته). سه مورد با تصمیمات
قبلی مستندشده تناقض داشتند و قبل از ثبت تسک از کاربر پرسیده شدند.

## ۲. بحث و تصمیمات

### موضوع ۱: تعریف فنی Agent (پیش‌زمینه، بدون تسک جدید)

- **بحث:** تعریف کامل (حلقه‌ی ادراک→تصمیم/برنامه‌ریزی→عمل + نقش + مهارت + ابزار + ارتباط
  بین Agentها + مشارکت در Workflow) همان روز در `00_VISION/06-glossary.md`'s Agent entry و
  `04-digital-employee.md` ثبت شد.
- **تصمیم:** بدون تغییر می‌ماند؛ خلأهای زیر روی همین تعریف ساخته می‌شوند.

### موضوع ۲: Tool-calling واقعی (معنای دقیق تئوری Agent)

- **بحث:** بررسی کد نشان داد در حال حاضر **هیچ native tool-calling/function-calling** در
  لایه‌ی LLM provider وجود ندارد (نه در `@o2n/llm-providers`، نه در `LlmService`). اجرای
  ابزارها (`routes/tools.ts`، Skillها، Pluginها) کاملاً جدا از حلقه‌ی چت، از طریق route‌های
  HTTP مجزا صدا زده می‌شود — یا مستقیم توسط کاربر/UI، یا توسط Workflow Engine (یک DAG جبری،
  نه تصمیم زنده‌ی مدل). یعنی مدل خودش وسط یک مکالمه تصمیم نمی‌گیرد «الان باید ابزار X را صدا
  بزنم».
- **تصمیم:** باید ساخته شود. مدل باید بتواند وسط یک نوبت چت تصمیم بگیرد یک Tool/Skill را
  صدا بزند، نتیجه را ببیند، و دوباره تصمیم بگیرد (الگوی ReAct/agentic loop). اجرای هر
  Tool/Skill باید به‌صورت یک رویداد قابل‌مشاهده در چت نمایش داده شود (نه فقط پاسخ نهایی)، و
  نتیجه‌اش در Memory ذخیره شود — دقیقاً طبق درخواست کاربر.

### موضوع ۳: محدودسازی ابزار/مهارت به Agent + درخواست از Agent دیگر

- **بحث:** محدودسازی خودِ دسترسی از قبل هست (`agent_skill_grants`/`agent_plugin_grants`،
  permission-gated — RT-026/RT-080). چیزی که نیست: وقتی یک Agent به مهارت/ابزاری نیاز دارد
  که خودش ندارد ولی Agent دیگری در همان سازمان دارد، باید بتواند به‌جای شکست ساده، از آن
  Agent درخواست انجام کار بدهد (delegation خودکار، نه فقط پیام دستی — `agent_messages`
  فعلی صرفاً یک صف پیام دستی است، نه یک تصمیم خودکار «من این را ندارم پس درخواست می‌دهم»).
- **تصمیم:** باید ساخته شود، به‌عنوان بخشی از همان agentic loop موضوع ۲.

### موضوع ۴: استاندارد Skill + فرم no-code

- **بحث:** کاربر خواست Skillها مطابق یک استاندارد «agent skill» تعریف شوند. فرم فعلی
  (`web/app/skills/page.tsx`) فقط دو نوع step پشتیبانی می‌کند (`Webhook`، `Telegram
message` — طبق `SkillStepSchema`'s v1 scope) و برای اینکه یک کارمند غیرفنی هم بتواند
  Skill بسازد کافی نیست.
- **سؤال باز، حل‌نشده:** کاربر دقیقاً کدام استاندارد را مدنظر دارد؟ (مثلاً MCP، یا یک قرارداد
  داخلی خودمان) — قبل از پیاده‌سازی باید مشخص شود.
- **تصمیم:** فرم no-code گسترش پیدا می‌کند (Step typeهای بیشتر: Plugin، Skill دیگر، Agent
  delegation)، منتظر تعیین استاندارد دقیق برای شکل‌گیری نهایی schema.

### موضوع ۵: زمان‌بندی (Schedule) یک Agent — ⚠️ تناقض با RT-007، حل شد

- **بحث اولیه:** کاربر گفت «یک Agent نباید به‌صورت Schedule اجرا شود» — این دقیقاً برعکس
  چیزی بود که `RT-007` ساخته (self-scheduling: یک Agent با `schedule.enabled=true` خودش را
  طبق `intervalMinutes` trigger می‌کند، `services/scheduler.ts`).
- **روشن‌سازی کاربر:** رفتار self-scheduling فعلی حذف نمی‌شود — **مدل Schedule باید غنی‌تر
  شود**: مشخص کردن اینکه کدام ابزار/Skill/Workflow در آن اجرای زمان‌بندی‌شده استفاده شود (نه
  فقط یک `prompt` متنی ثابت طبق schema فعلی)، و پشتیبانی از الگوهای ساعت/روز/ماه (نه فقط
  `intervalMinutes` ساده). به‌علاوه، سیستم باید یک مفهوم **timezone** انتخاب‌شدنی داشته باشد
  که الان اصلاً وجود ندارد (نه روی `organizations`، نه جای دیگری — چک شد).
- **تصمیم:** `AgentScheduleSchema` گسترش پیدا می‌کند (هدف اجرا: tool/skill/workflow + الگوی
  cron-مانند ساعت/روز/ماه) + یک ستون/تنظیم timezone اضافه می‌شود (به‌احتمال زیاد در سطح
  سازمان، مشابه الگوی زبان در RT-083 — پیش‌فرض سازمانی + احتمال override).

### موضوع ۶: تنظیمات LLM Provider

- **بحث:** سه ادعا بررسی شد. (۱) apiKey برای provider=ollama هم در فرم `/settings` و هم در
  schema بک‌اند (`LlmConfigSetSchema`) اجباری است (`required`/`min(1)`) — **باگ واقعی**،
  چون ollama نیازی به کلید ندارد. (۲) فیلد `model` یک `<input>` متنی آزاد است، نه لیستی از
  مدل‌های واقعی همان provider — **خلأ واقعی**. (۳) provider فقط یک‌بار در سطح سازمان تنظیم
  می‌شود (`llm_configs`، یک ردیف به‌ازای هر org)، انتخاب per-chat وجود ندارد.
- **⚠️ تناقض یافته‌شده و حل‌شده:** بند (۳) دقیقاً همان چیزی است که در بازبینی
  Phase-۰ روادمپ (۲۰۲۶-۰۷-۱۱، `DONE.md`) عمداً به Control Plane واگذار شده بود («Model
  Router چند-مدلی همزمان»). **کاربر تأیید کرد این تصمیم قبلی همچنان درست است** — بند (۳) از
  scope این batch حذف شد؛ فقط (۱) و (۲) به‌عنوان تسک ثبت می‌شوند.

### موضوع ۷: استریم/عملیات real-time — WebSocket — ⚠️ تناقض با تصمیم ۲۰۲۶-۰۷-۱۱، معکوس شد

- **بحث:** تصمیم مستندشده‌ی قبلی (`DONE.md`، «Roadmap Phase ۰ audit»، ۲۰۲۶-۰۷-۱۱) این بود
  که SSE جایگزین کافی WebSocket برای استریم چت است، و **کاربر همان موقع صریحاً تأییدش کرده
  بود** («کاربر تأیید کرد این تصمیم قبلی درسته و نیازی به کار جدید نیست»).
- **تصمیم امروز کاربر: این تصمیم معکوس می‌شود.** WebSocket باید جایگزین SSE شود — طبق گفته‌ی
  کاربر «هر وب‌سوکتی که سریع‌تر عمل می‌کند». دامنه: هم استریم پاسخ چت، هم هر رویداد
  real-time دیگری که از موضوع ۲ (نمایش زنده‌ی اجرای Tool/Skill در چت) نیاز می‌شود.

### موضوع ۸: منوهای چندزبانه (i18n کامل، نه فقط چت)

- **بحث:** کاربر انتظار داشت با تغییر زبان (مثلاً فارسی)، کل منوها فارسی شوند. بررسی نشان
  داد این باگ نیست — محدودیت **مستندشده‌ی خودِ RT-083** است (نوشته شده در `DONE.md`: «عمداً
  محدود به همین صفحه [چت]، نه کل اپ ... صفحات دیگه هنوز LTR انگلیسی‌محورن — یک گسترش
  آینده»). بقیه‌ی صفحات (Skills, Marketplace, Users, ...) هنوز رشته‌های انگلیسی سخت‌کد شده
  دارند.
- **تصمیم:** گسترش i18n به کل UI (نه فقط چت/تنظیمات) به‌عنوان یک تسک جدا ثبت می‌شود.

### موضوع ۹: فعال‌سازی اولین اجرا + امنیت ارتباط با Marketplace/Control-Plane

- **بحث:** الان activation فقط یک env var دستی (`ACTIVATION_KEY`) است که باید قبل از اجرای
  gateway در `.env` تنظیم شود — هیچ صفحه/UI برای وارد کردنش در اولین اجرا وجود ندارد (چک
  شد: هیچ فایلی زیر `web/app` با نام مرتبط به activation پیدا نشد). ارتباط با Marketplace هم
  از یک secret کاملاً جدا و ثابت (`MARKETPLACE_API_KEY`) استفاده می‌کند که هیچ ربطی به
  تأیید activation code ندارد.
- **تصمیم:** (۱) یک صفحه‌ی first-run برای وارد کردن activation code ساخته شود. (۲) بعد از
  تأیید موفق کد (از طریق Control Plane)، یک توکن امنیتی مخصوص تولید/ذخیره شود که برای تمام
  ارتباطات بعدی با Marketplace/Control-Plane استفاده شود (به‌جای secretهای ثابت پراکنده‌ی
  فعلی). طبق پیشنهاد کاربر، یک فایل کانفیگ JSON مشترک تنظیمات Marketplace/Control-Plane را
  نگه می‌دارد. **این یک تغییر سه‌پلینی است** (Runtime می‌سازد/مصرف می‌کند، Control Plane باید
  توکن را صادر کند، Marketplace باید بپذیردش) — جزئیات دقیق پروتکل نیاز به یک تصمیم معماری
  مشترک دیگر دارد، این‌جا فقط طراحی/شروع کار به‌عنوان تسک Control Plane (که مالک activation
  است) ثبت می‌شود؛ سمت Runtime/Marketplace بعد از مشخص‌شدن پروتکل دنبال می‌شود.

## ۴. نکات آزاد

- تصمیم ۲۰۲۶-۰۷-۱۱ درباره‌ی «Model Router چند-مدلی همزمان» (انتخاب provider هنگام چت) دوباره
  بررسی و **تأیید مجدد** شد — همچنان کار Control Plane است، نه Runtime.
- تصمیم ۲۰۲۶-۰۷-۱۱ درباره‌ی WebSocket **معکوس** شد (بالا، موضوع ۷) — یک نمونه‌ی واقعی از
  اینکه تصمیمات ADR-مانند این پروژه می‌توانند با گذر زمان و context بیشتر تجدیدنظر شوند؛ اگر
  بعداً باز هم این حوزه بررسی شد، هر دو تصمیم (۲۰۲۶-۰۷-۱۱ و این یکی) باید در نظر گرفته شوند.
