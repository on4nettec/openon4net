# جلسه ۵ — ۲۰۲۶-۰۷-۱۷

**حضور:** کاربر (owner)، Claude
**مدت:** —
**موضوع:** مدل کامل ثبت‌نام/activation در Control Plane + تفاوت شخصی/سازمانی + چندزبانگی (i18n)

---

## ۱. دستور کار (Agenda)

این جلسه با یک سوال درباره‌ی چندزبانگی (RTL/LTR در چت) شروع شد و به یک
توضیح کامل از مدل ثبت‌نام/activation کشیده شد. هر دو موضوع اینجا ثبت
می‌شوند چون هر دو صرفاً **بررسی وضعیت موجود** بودند، نه شروع پیاده‌سازی.

### بخش الف: چندزبانگی (i18n) در چت

کاربر پرسید آیا RTL/LTR درست نمایش داده می‌شه. بررسی نشون داد **هیچ‌کدوم
از این‌ها وجود نداره**، نه در Runtime نه در Control Plane:

- هیچ کتابخانه i18n، هیچ فایل JSON locale، هیچ language switcher.
- `<html lang="en">` هاردکد در هر دو وب‌اپ.
- هیچ ستون `language`/`locale` روی `users` یا `organizations`.
- صفحه چت تک‌پنله (نه ۳ پنل Control/Chat/Workspace طبق سند مرجع).

کاربر سپس یک مدل کامل توضیح داد (بخش ب پایین) — چندزبانگی ادغام شد توی
همون بحث، چون طراحی زبان (پیش‌فرض سازمان + override کاربر) به مدل
کاربر/سازمان همون activation system وابسته‌ست.

### بخش ب: مدل کامل ثبت‌نام و Activation

کاربر این جریان رو توضیح داد:

۱. کاربر Runtime رو یا شخصی یا سازمانی هاست می‌کنه. برای این کار به
`control.on4net.com` می‌ره، یک حساب می‌سازه (این خودِ سیستم Control
Plane هست) و یک activation code دریافت می‌کنه.
۲. کاربر مجاز به دریافت **۵ activation code** هست (سقف در سطح حساب).
۳. بعد از ثبت‌نام، کاربر مشخص می‌کنه این activation code برای **شخصی**
هست یا **سازمانی**:

- **شخصی:** در Runtime فقط مجاز به ساخت **یک** کاربر؛ بخش دسترسی
  Agentها به کاربران (Agent Access) کاملاً غیرفعال می‌شه.
- **سازمانی:** تا **۳ کاربر رایگان**؛ از کاربر چهارم به بعد پولی می‌شه.
- یعنی activation مشخص می‌کنه Runtime چطور نصب/پیکربندی بشه.
  ۴. برای این‌که کاربر مجبور نشه چند جا جدا ثبت‌نام کنه، **مارکت‌پلیس به‌شکل
  micro-frontend داخل Control Plane نمایش داده می‌شه**. وقتی کاربر
  activation خودش رو در Runtime فعال می‌کنه، دسترسی استفاده از مارکت‌پلیس
  براش باز می‌شه — می‌تونه مثل وردپرس یا از مارکت‌پلیس پلاگین نصب کنه یا
  خودش آپلود کنه.
  ۵. هنگام دریافت activation، یک سری مشخصات دیگه هم گرفته می‌شه.
  ۶. یک مسیر جدا هم هست: کاربر در Control Plane می‌تونه یک activation
  **سازمانی/شرکتی** برای یک شرکت/سازمان بسازه — این یک بخش verification
  داره که **صریحاً به فاز بعدی موکول شد**، اینجا فقط ثبت می‌شه که وجود
  داره، ساخته نمی‌شه.

## ۲. بررسی وضعیت موجود (قبل از هر تصمیم)

بررسی مستقیم کد نشون داد **هیچ‌کدوم از این مدل هنوز وجود نداره**:

- **ثبت‌نام خودکار در Control Plane اصلاً وجود نداره.** صدور
  activation key امروز یک عمل کاملاً دستی/فقط-اپراتوره
  (`routes/admin-activation.ts`'s کامنت صریح: _"self-serve signup is
  out of scope for MVP-lite"_). صفحه `/admin` فقط یک فرم paste-کردن
  `ADMIN_API_KEY` هست، نه لاگین واقعی.
- **Control Plane اصلاً جدول `users` نداره** — فقط `organizations`،
  `activation_keys`، `wallets`، `credit_transactions`.
- `activation_keys` هیچ ستون نوع (شخصی/سازمانی) و هیچ سقف seat/user
  نداره. `organizations.plan` فقط یک tier قیمتی (starter/team/business/
  enterprise) هست، ربطی به شخصی/سازمانی نداره.
- **Runtime هیچ مفهوم «حالت شخصی» نداره.** `agent_access_bindings`
  (RT-024) کاملاً مستقل از activation/plan کار می‌کنه — activation-client
  فقط `organizationId/plan/status/featureFlags` رو می‌گیره، نه نوع
  activation.
- هیچ منطق «۳ کاربر رایگان، بعد پولی» جایی وجود نداره.
- **مارکت‌پلیس اصلاً وب‌سایت/frontend نداره** — فقط یک backend service.
  هیچ الگوی micro-frontend (module federation/iframe/rewrite) هم جایی
  در کل مونوریپو پیاده نشده.

## ۳. تصمیمات (با پرسش/پاسخ همین جلسه)

### موضوع الف: روش احراز هویت ثبت‌نام

- **تصمیم:** ایمیل/پسورد + مگ‌لینک تأیید — همون الگوی موجود Runtime's
  invitation system (hashed-token magic-link)، نه یک سرویس ایمیل جدید.

### موضوع ب: معنای «مارکت‌پلیس به‌شکل micro-frontend»

- **بحث:** چون مارکت‌پلیس اصلاً وب‌سایت نداره، «micro-frontend واقعی»
  یعنی باید از صفر یک frontend کامل برای Marketplace ساخته بشه، به‌علاوه
  یک زیرساخت micro-frontend که هیچ‌جای این مونوریپو سابقه نداره.
- **تصمیم:** به‌جاش، Control Plane مستقیم API مارکت‌پلیس رو صدا می‌زنه و
  خودش UI می‌سازه — دقیقاً همون الگویی که Runtime همین الان داره
  (`services/marketplace-client.ts` + `routes/marketplace.ts` +
  `web/app/marketplace/page.tsx`). بدون frontend جدید برای Marketplace،
  بدون زیرساخت micro-frontend جدید.

### موضوع ج: غیرفعال‌سازی Agent Access در حالت شخصی

- **تصمیم:** کل UI/routeهای Agent Access مخفی/غیرفعال بشه (نه این‌که
  فقط منطق grant دست‌نخورده بمونه) — چون در حالت شخصی فقط یک کاربر هست،
  خود مفهوم «دسترسی چند کاربر به Agent» بی‌معنیه.

### موضوع د: دامنه‌ی «۳ کاربر رایگان، بعد پولی»

- **تصمیم:** فقط شمارش/محدودیت ساخته بشه (رد کردن دعوت کاربر پنجم و
  بعدش)، بدون درگاه پرداخت واقعی — هم‌راستا با محدودیت شناخته‌شده‌ی
  CP-013 (billing واقعی هنوز عمداً معلقه).

### موضوع ه: چندزبانگی (از بخش الف agenda)

- **تصمیم (بر اساس صحبت کاربر، هنوز به تسک رسمی تبدیل نشده):** فایل‌های
  JSON key/value با `en.json` به‌عنوان مرجع؛ اولین ورود کاربر یک انتخاب
  زبان نشون می‌ده؛ اگه زبان انتخابی غیر از انگلیسی بود و فایلش نبود، با
  AI ساخته و اضافه می‌شه. **پیش‌فرض زبان در سطح سازمان** ذخیره می‌شه، ولی
  **هر کاربر می‌تونه override شخصی خودش رو داشته باشه** (مثال کاربر:
  سازمان عربی، یک کاربر فرانسوی توی همون سازمان). چون Control Plane
  جدول `users` نداره، این override فقط در سطح **Runtime** معنی داره
  (Control Plane حداکثر یک زبان پیش‌فرض سازمانی می‌تونه داشته باشه).
  **این بخش هنوز تسک رسمی نشده** — باید در یک جلسه/batch جدا دقیق‌تر
  طراحی بشه (مکانیزم AI-generation، کش فایل‌های ساخته‌شده، و غیره).

## ۵. نکات آزاد

- این جلسه فقط **تصمیمات و تسک‌بندی** رو ثبت می‌کنه؛ هیچ‌کدوم از
  CP-025..027 یا RT-081/082 هنوز شروع نشدن — برخلاف batch پلاگین جلسه ۴
  (که کاربر صریحاً گفت بدون سوال جلو برم)، برای این batch هنوز دستور
  صریح شروع کار داده نشده — منتظر تأیید کاربر می‌مونیم.
- زنجیره‌ی وابستگی: CP-026 به CP-025 وابسته‌ست (باید کاربر لاگین‌شده
  داشته باشیم تا بشه activation key بهش نسبت داد). RT-081/082 به CP-026
  وابسته‌ن (باید نوع/سقف activation از Control Plane بیاد). CP-027 فقط
  به CP-025 وابسته‌ست (کاربر لاگین‌شده)، نه به CP-026.
- CP-025 خودش یک زیرساخت احراز هویت کاملاً جدیده (رمزنگاری پسورد،
  session/JWT، ایمیل مگ‌لینک) — سطح امنیتی حساسی داره، پس شایسته‌ی
  بررسی دقیق‌تر قبل از شروعه، نه فقط یک batch سریع.
