## ۱. دستور کار (Agenda)

کاربر شکل کامل اکوسیستم Plugin رو یک‌جا توضیح داد و خواست قبل از هر کدنویسی،
این توی مستندات ثبت بشه و تسک‌های لازم مشخص بشن. نکات مطرح‌شده:

۱. همه‌ی ابزارها به‌صورت Plugin نصب می‌شوند.
۲. Pluginی که نیاز به دیتابیس داره، باید طوری دیپلوی بشه که دیتابیس مرکزی
بزرگ نشه — تشبیه شد به «مثل میکروسرویس روی Runtime».
۳. بعضی Pluginها اصلاً دیتابیس لازم ندارند — فقط یک provider/wrapper نازک
دور یک سرویس بیرونی‌اند، به‌عنوان tool یا داخل Workflow قابل‌استفاده.
۴. Plugin رو یا برنامه‌نویس انسانی می‌سازه یا یک «ایجنت برنامه‌نویس».
۵. Marketplace هم برای Plugin و هم برای Skill دو ردیف دارد: رایگان / فروشی.
۶. بخش sandbox برای **تست** Plugin قبل از production است؛ فقط بعد از رد
شدن از این گیت، Plugin قابل عرضه در فروشگاه می‌شود.
۷. Pluginهایی که ایجنت تولید می‌کند مستقیم در Runtime اجرا می‌شوند.
۸. نصب از Marketplace اجباری نیست — کاربر self-hosted می‌تواند Plugin خودش
را بسازد و مستقیم استفاده کند؛ فقط اگر بخواهد **بفروشد**، باید از
sandbox رد شود (هم به‌خاطر امنیت).
۹. Pluginها باید دسته‌بندی (category) مشخص و قابل‌جستجو داشته باشند.
۱۰. چون Workflow داریم، بعضی Pluginها باید به‌عنوان step قابل‌استفاده باشند.
۱۱. برای امنیت Agentها، باید بشه دسترسی یک Agent به یک Plugin خاص را
grant/deny کرد (مشابه چیزی که برای Skills داریم).

سوال مستقیم کاربر: آیا نیاز به یک **استاندارد رسمی Plugin** و/یا **CLI**
داریم؟

## ۲. بررسی وضعیت موجود (قبل از هر تصمیم جدید)

- `packages/plugin-sdk` (`@o2n/plugin-sdk`) و `packages/create-o2n-plugin`
  از قبل ساخته شده‌اند (جلسه ۲، RT-037). این **استاندارد و CLI** همون
  چیزیه که کاربر پرسید — از قبل هست، نیاز به ساخت از صفر نیست.
  - محدودیت مستندشده در خود کد (`plugin-sdk/src/index.ts`): فقط
    manifest/type-building است؛ _"Runtime does not load or run a built
    Plugin yet"_.
- `docs/spect/02_ARCHITECTURE/09-plugin-sandbox.md` از قبل یک روادمپ
  ۳سطحی تعریف کرده: **Level 0** (in-process، فقط first-party — همینی که
  الان MVP هست)، **Level 1** (process جدا)، **Level 2** (WASM — هدف
  نهایی برای third-party، عمداً معلق مونده).
- جدول `plugins` (Marketplace، `0001_marketplace_core.sql`) هیچ ستون
  `category` ندارد.
- الگوی per-agent grant برای Skills از قبل کامل هست
  (`agent_skill_grants` + `SkillGrantService` + `POST/DELETE
/v1/agents/:id/skills/:skillId/grant`) — قابل‌تکرار عیناً برای Plugin.
- Workflow Engine فعلاً ۵ نوع step دارد: `agent`/`tool`/`human`/
  `parallel`/`condition` (`workflow-executor.ts`) — بدون نوع `plugin`.
- مدل رایگان/فروشی Marketplace از قبل به‌عنوان ADR-012 ثبت شده
  («activation-gated free + purchase-gated paid») و طبق جلسه ۲ قرار بود
  هم روی Plugin هم روی Skill اعمال بشه.

## ۳. بحث و تصمیمات (با پرسش/پاسخ همین جلسه)

### موضوع الف: توپولوژی دیپلوی Plugin دیتابیس‌دار

- **بحث:** آیا «مثل میکروسرویس روی Runtime» یعنی Runtime باید یک زیرسیستم
  جدید process/container-orchestration بسازد (هر Plugin = deployment جدا
  با DB خودش)، یا یعنی چیز سبک‌تری؟
- **تصمیم:** Plugin همچنان **in-process یا WASM-sandboxed** اجرا می‌شود
  (طبق روادمپ موجود Level 0/1/2 در `09-plugin-sandbox.md`)، ولی هر
  Pluginی که نیاز به ذخیره‌سازی دارد، یک **schema/namespace جدا** داخل
  همون Postgres مشترک می‌گیرد — نه یک deployment/container کاملاً مجزا.
  این یعنی **بدون نیاز به ساخت یک زیرسیستم orchestration جدید**؛ فقط یک
  لایه‌ی provisioning ایزوله‌ی schema لازم است. این همان چیزی است که
  «دیتابیس مرکزی بزرگ نشود» را برآورده می‌کند، بدون هزینه‌ی معماری یک
  mini-PaaS.

### موضوع ب: معنای «sandbox برای تست»

- **بحث:** آیا این یعنی WASM sandbox واقعی (Level 2) که باید همیشه فعال
  باشد، یا یک گیت یک‌بارمصرف پیش از انتشار؟
- **تصمیم:** یک **گیت یک‌بارمصرف پیش از انتشار** — Plugin در یک محیط
  ایزوله تست/اجرا می‌شود، اگر رد شد وضعیتش به «قابل عرضه» تغییر می‌کند.
  سطح اجرای همیشگی (runtime execution) همچنان همان Level 0 فعلی می‌ماند؛
  ساخت واقعی WASM sandbox (Level 2) به‌صراحت **معلق** می‌ماند مگر بعداً
  صراحتاً درخواست شود.

### موضوع ج: پلاگین‌های تولیدشده توسط ایجنت

- **بحث:** آیا این Pluginها کاملاً بدون بررسی اجرا می‌شوند یا فقط از یک
  بخش خاص معاف‌اند؟
- **تصمیم:** فقط از **فرایند انتشار/sandbox-گیت Marketplace** معاف‌اند —
  مسیر امنیت اجرا (permission/budget/audit، همان Level 0) فرقی نمی‌کند.
  یعنی «مستقیم در Runtime اجرا شدن» به‌معنای دور زدن چک‌های اجرایی نیست،
  فقط به‌معنای نبودن مرحله‌ی «انتشار در فروشگاه» است.

### موضوع د: پیش‌نیاز ایجنت برنامه‌نویس — AI Gateway

- **بحث (تکمیل بعدی کاربر):** ایجنت برنامه‌نویس (که Plugin تولید می‌کند)
  فقط زمانی قابل‌استفاده است که از **AI Gateway سمت Control Plane** برای
  فراخوانی LLM استفاده کند — نه اتصال مستقیم به یک provider.
- **تصمیم:** این یعنی قابلیت «ایجنت Plugin می‌سازد» به **CP-012 (Managed
  AI Gateway)** وابسته است — که در حال حاضر در
  `TODO-openon4net-platform.md`'s بخش C (تصمیم معماری بزرگ، هنوز
  ساخته‌نشده) قرار دارد. تا CP-012 ساخته نشود، مسیر اجرای مستقیم
  Plugin توسط ایجنت برنامه‌نویس **بلاک** است — نه به‌خاطر محدودیت فنی
  دیگر، بلکه چون این پیش‌نیاز صریح کاربر است.

### موضوع ه: دسته‌بندی، Workflow step، per-agent grant

- **تصمیم:** هر سه به‌صورت مستقیم پذیرفته شدند؛ الگوی موجود Skills برای
  per-agent grant عیناً روی Plugin تکرار می‌شود. دسته‌بندی به‌عنوان یک
  ستون/جدول جدید در Marketplace اضافه می‌شود. Workflow یک نوع step جدید
  (`type: 'plugin'`) می‌گیرد.

### موضوع و: مدل رایگان/فروشی برای Skill هم

- **تصمیم:** طبق ADR-012 این قبلاً تصمیم‌گیری شده بود که هم روی Plugin
  هم روی Skill اعمال بشه؛ این جلسه فقط تأیید کرد و یک تسک تأیید/تکمیل
  (MKT-026) برایش ثبت شد که واقعاً کامل پیاده شده یا خیر.

  یک وابستگی صریح بین‌پلین است، نه یک محدودیت فنی قابل‌دورزدن.
- ساخت واقعی WASM Sandbox (Level 2) هنوز به‌صراحت خارج scope این دسته از
  تسک‌هاست — تصمیم این جلسه بود که سطح فعلی (Level 0 + سند schema
  ایزوله + یک گیت تست یک‌بارمصرف) برای این فاز کافی است.

  — این جزئیات باید هنگام طراحی RT-078 مشخص شود، وقتی CP-012 آماده شد.
